Build Walkthrough

This page is a behind-the-scenes look at how I built my online resume site using HTML, CSS, and JavaScript. I walk through my structure, styling decisions, and small interactive features to show that I understand how everything fits together.

Overview

This walkthrough shows how each part of my resume site is built. For every screenshot, the top half shows code in VS Code and the bottom half shows the matching UI in the browser. I explain the HTML structure, CSS classes, and JavaScript behavior so it is clear that I understand how the site works end-to-end.

How I Structured the HTML

I started by planning the structure in HTML. I wanted each major part of the resume (navigation, header, objective, skills, experience, education, projects, contact, and footer) to live in its own clearly labeled section. That makes the site easier to maintain, and it also helps screen readers and assistive technology understand the page.

Navigation Bar

I use a semantic <nav> element with anchor links that point to section IDs like #about, #skills, and #projects. The links are wrapped in a .navwrap container so I can center them and control their spacing with flexbox. Using IDs for each section makes the navigation feel snappy and keeps everything on a single page.

Header, Photo, and Quick Facts

The header introduces who I am at a glance. I group my profile image and text in a container with the .id class, which uses flexbox to place them side by side. The avatar is an <img> element with the .avatar class so it stays circular. Under my name, I include a short summary line and a row of .chip elements that highlight quick facts like Dean’s List and location.

Objective Section

My objective lives inside <section id="about"> so the nav link for “About” scrolls directly here. I keep this section simple: one .card that stretches across the width of the content and clearly states what kinds of roles I’m looking for and when I’m available. I use <strong> tags to call out important keywords without over-formatting the text.

Skills Layout

For the Skills section, I wanted to balance technical skills and more narrative highlights. I wrap the two sides in a .grid container so they appear as two cards side by side on larger screens. The left card breaks skills into categories like Programming Languages and Tools, and the right card uses a bullet list to highlight hands-on experience and communication skills. On smaller screens, the grid collapses to one column so the content remains readable.

Work Experience

The Experience section follows a similar pattern: each job is wrapped in a .card, and I use .row to display the position, organization, dates, and location on a single line. I then list responsibilities and achievements in bullet points. To keep the bullets visually consistent, I use a .list-item wrapper with a small decorative .dot span before the text.

Education

My Education section is a single full-width .card that highlights my degree at RIT. I separate the school and location into one .row, and my degree, expected graduation, and Dean’s List recognition into another. Below that, I summarize key coursework in one bullet point so that hiring managers can quickly see the technical foundation behind the rest of the site.

Projects – Password Strength Checker

The Projects section currently focuses on my Password Strength Checker demo. The project content is wrapped in .card.project. I use a .title div for the project name, a short description, and then an <a> element styled as a primary button that links out to the demo at demos/password/index.html. This shows that I can connect a standalone tool into a larger portfolio site.

Contact

For the Contact section, I wanted something simple and printable. I keep everything in one sentence inside a .card, using <strong> labels for Phone, Email, and LinkedIn, followed by clickable links. I use a tel: link for my phone, a mailto: link for my email, and a normal https:// link for LinkedIn, so the contact information works well on both desktop and mobile.

Footer

The footer is short on purpose. It includes a line with my name, a dynamic year (controlled with JavaScript), and a quick summary that the site is accessible, responsive, print-ready, and supports dark mode. The year itself lives in a <span id="year">, which I update when the page loads so I never have to hardcode the date again.

Adding Interactivity with JavaScript

After I was happy with the HTML layout, I added a small amount of JavaScript to make the site feel more interactive. My goals here were to keep the script lightweight and focused, while still demonstrating that I know how to work with the DOM, events, and browser storage.

Theme Toggle and Dynamic Year

This inline script handles two main behaviors:

  • Dark/Light Mode Toggle: I read the current theme preference from localStorage (or default to light mode), then set a data-theme attribute on the <html> element. When the user clicks the theme toggle button, I switch the value, update the attribute, save the new theme back to localStorage, and change the button label so it always matches the active mode.
  • Dynamic Footer Year: I grab the #year span and set its textContent to new Date().getFullYear(). It’s a small detail, but it shows that the site is current without needing manual edits every January.

Even though this is a small amount of JavaScript, it ties the UI, CSS variables, and user preferences together in a meaningful way.

How I Designed the CSS

I treated the stylesheet like a small design system. Instead of styling everything one-off, I built reusable pieces: theme variables, layout primitives, cards, grids, buttons, chips, and so on. That makes the site easier to update and keeps the look consistent across pages.

Theme Variables

At the top of my stylesheet, I define CSS custom properties (variables) for colors like backgrounds, text, borders, and accents. I set a default set on :root for light mode and override them inside html[data-theme="dark"] for dark mode. Because the rest of the CSS uses these variables, I can switch themes by changing just one attribute on the root element.

Global Resets and Base Styles

Here I apply a reset so every element uses box-sizing: border-box, which makes width and padding much easier to reason about. I also remove the default body margin, pick a system UI font stack, and wire the base text and background colors to my theme variables. Link colors and hover states are also defined here so they’re consistent everywhere.

Page Layout and Header

The .container class limits the content width and centers it with auto margins, while the header uses flexbox to position my photo and text side by side. The .id class groups the avatar and name, and .avatar makes the image a circle and crops it nicely with object-fit: cover. This keeps the top of the page visually balanced and easy to scan.

Buttons

Buttons are built using a base .btn class and an optional .primary modifier. The base class handles padding, borders, rounded corners, and flex alignment so icons and text can sit nicely on one line. .btn.primary applies the accent color and white text, which I reuse for the Download PDF and Password Checker buttons. The .btns container lines up multiple buttons with consistent spacing.

Navigation Styling

The navigation bar is styled to be sticky, so it stays at the top of the viewport when the user scrolls. The nav element gets a background tied to the theme and a light bottom border. Inside, .navwrap uses flexbox to horizontally align links, with gap spacing and horizontal scrolling on very small screens. Each link has padding and rounded corners, and the hover state uses the card background color for a subtle highlight.

Section Spacing and Headings

To keep the page from feeling cramped, every section has vertical padding and a light border at the bottom. This visually separates each part of the resume. Headings like h2 and h3 get consistent sizes and margins so the hierarchy is clear. The .muted class uses a softer color for supporting text, such as subtitles and labels.

Grid Layout

The .grid class is my main layout tool for side-by-side cards in the Skills and Experience sections. On larger screens, it uses two equal-width columns with a gap. A media query at max-width: 820px switches the grid to a single column, which keeps the content readable on tablets and phones without needing a totally different layout.

Card Components

The .card class is the backbone of my design system. It gives each block a consistent background, border, border radius, and padding. The .project card extends this by using flexbox to stack the project title, description, and button with comfortable spacing. This shared “card” look ties the whole site together visually.

Chips

I use the .chips and .chip classes to create the pill-shaped labels under my name in the header. .chips is a flex container that wraps when there isn’t enough horizontal space. Each .chip has a background, rounded corners, border, and compact typography, making it a reusable pattern for quick metadata like location or honors.

Form Fields

Even though the live resume doesn’t currently have a full contact form, I set up a .field pattern for potential future use. It includes a bold label on its own line and an input or textarea with full width, padding, rounded corners, and theme-aware colors. This keeps the styling consistent if I decide to add forms later.

Table Styling

I created a small table design system for use in demos, such as an SQL dashboard mockup. The .table-wrap container enables horizontal scrolling and adds rounded corners. Table cells use consistent padding and borders, and the header row in thead is sticky so column labels stay visible when scrolling. The .badge, .badge.ok, and .badge.warn classes turn small pieces of text into status pills.

Images Inside Cards

For this walkthrough page, I wanted screenshots to look clean and consistent. The image rules make sure that any <img> inside a .card scales to 100% of the card width, keeps its aspect ratio, and has rounded corners and a subtle border. I also give each image a margin at the bottom so the caption text doesn’t sit directly against it.

Print Styles

Finally, I use a @media print block to make sure the resume prints cleanly. I hide navigation, interactive buttons, and the theme toggle so only the content remains. The colors switch to a white background with black text, card borders are softened, and link styling is removed. This makes the printed version feel closer to a traditional resume while still using the same HTML.
